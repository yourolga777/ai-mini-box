# Спецификация: Почтовый плагин — Рекомендации после тестирования (v1)

**Разработчик:** Почтовый плагин  
**Приоритет:** P2  
**Статус:** Рекомендации (не баги)

---

## Результат анализа

Критических багов в почтовом плагине не обнаружено. Email-сообщения корректно загружаются в базу знаний (`KnowledgeBase`) и отображаются в интерфейсе.

---

## Рекомендация: Индексация Email в поиске сообщений

**Проблема:** Поиск в `Messages.tsx` (`MessageSearchBar`) ищет по `MessageModel` в БД. Email-сообщения могут храниться отдельно (например, в `KnowledgeBase`) и не участвовать в общем поиске.

**Что сделать:**
- Убедиться, что email-сообщения сохраняются в `MessageModel` (с полем `source = "email"`)
- Или добавить поиск по `KnowledgeBase` параллельно с поиском по сообщениям

**Acceptance criteria:**
- Email-сообщения отображаются в общем списке сообщений с источником "Email"
- Поиск по тексту находит в том числе email-сообщения

---

## Границы ответственности

- Не менять фронтенд
- Не менять другие плагины
- Изменения опциональны

---

## Заключение (закрыто)

**Статус:** Рекомендация выполнена архитектурой по умолчанию.

| Пункт | Статус | Обоснование |
|---|---|---|
| Сохранение в `MessageModel` с `source="email"` | ✅ Не требует изменений | `MessageSource.EMAIL` уже в core; плагин пишет как Telegram |
| Отображение в общем списке | ✅ Не требует изменений | `GET /messages` не фильтрует по `source` |
| Поиск по тексту находит email | ✅ Не требует изменений | `search()` ищет по полю `text` без фильтрации |
| Не менять фронтенд | ✅ Соблюдено | UI уже показывает колонку «Источник» |
| Не менять другие плагины | ✅ Соблюдено | Reply — внутри email-плагина |

**Дополнительно:** При сохранении тела письма очищать HTML от тегов, чтобы поиск по plain text работал корректно. Это задача email-плагина при реализации `message_handler.py`.
