# Тестирование сервисов AI mini box

## Три уровня тестирования

Каждый сервис тестируется на 3 уровнях:

---

### Уровень 1 — Unit-тесты (изолированно, без каркаса)

Разработчик тестирует логику сервиса с заглушками вместо каркаса.

```python
# tests/test_contacts.py
def test_add_contact():
    repo = MockContactRepo()             # заглушка вместо БД
    contact = Contact(name="Иван", phone="+7...")
    result = repo.add(contact)
    assert result.name == "Иван"
```

- `MockRepo` предоставляется каркасом (`ai_mini_box.tests.mocks`)
- Не требует БД, конфига, внешних зависимостей
- Запуск: `pytest tests/test_contacts.py` (внутри пакета сервиса)

---

### Уровень 2 — Интеграционные тесты (с каркасом)

Проверка, что сервис работает с реальным каркасом (тестовая БД SQLite `:memory:`).

```python
# tests/integration/test_contacts_integration.py
def test_contacts_via_cli(cli_runner, tmp_config):
    # cli_runner и tmp_config — фикстуры из conftest.py каркаса
    result = cli_runner.invoke(app, ["contacts", "add", "--name", "Иван"])
    assert result.exit_code == 0
    assert "Contact added" in result.output
```

**Каркас предоставляет (в `ai_mini_box/tests/conftest.py`):**
- `cli_runner` — Typer CliRunner с app
- `tmp_config` — временный config.json
- `mock_contact_repo`, `mock_product_repo`, `mock_message_repo` — mock-репозитории
- `tmp_path` — временная директория (built-in pytest)

- Внешние сервисы (IMAP, Telegram API, LLM) мокаются разработчиком сервиса
- Запуск: `pytest tests/integration/`

---

### Уровень 3 — E2E / Smoke-тесты (полная сборка)

Запускаются в CI перед мержем (GitHub Actions / GitLab CI).

```yaml
# .github/workflows/test.yml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - pip install -e "packages/core"          # ядро
      - pip install -e "packages/contacts"       # сервис A
      - pip install -e "packages/products"       # сервис B
      - pip install -e "packages/*"              # все остальные
      - pytest packages/                         # тесты ВСЕХ сервисов
      - ai-mini-box --help                       # smoke: все команды видны
      - ai-mini-box config check                 # smoke: конфиг работает
```

---

## CI-пайплайн — защита от поломок

Перед мержем любого PR прогоняются тесты ВСЕХ сервисов:

```
PR в репозиторий
       │
       ▼
CI запускает:
  ├─ unit-тесты сервиса A (изменённый)
  ├─ unit-тесты сервиса B
  ├─ unit-тесты сервиса C
  ├─ интеграционные тесты ВСЕХ сервисов
  └─ smoke test (CLI --help, config check)
       │
       ▼
Только всё зелёное → мёрж
```

---

## Роли

| Кто | Что делает | Как проверяет |
|---|---|---|
| Разработчик сервиса | Пишет код + unit-тесты | `pytest` локально, mocks |
| CI/CD | Интеграция + smoke | Все тесты + `ai-mini-box --help` |
| Каркас | Предоставляет фикстуры, mock-классы, CliRunner | `tests/conftest.py` |

## Структура тестов (монорепозиторий)

```
packages/
├── core/                          # каркас
│   └── ai_mini_box/tests/
│       ├── conftest.py            # общие фикстуры для ВСЕХ сервисов
│       ├── mocks.py               # MockContactRepo, MockProductRepo, MockMessageRepo
│       └── test_registry.py       # smoke: проверка авто-регистрации
│
├── contacts/                      # сервис контактов
│   └── tests/
│       ├── test_contacts.py       # unit-тесты (с MockContactRepo)
│       └── integration/
│           └── test_contacts_integration.py
│
├── products/                      # сервис товаров
│   └── tests/
│       ├── test_products.py
│       └── integration/
│           └── test_products_integration.py
│
└── ...                            # каждый сервис = своя папка tests/
```

## Правила для разработчика сервиса

1. **Unit-тесты** (`tests/test_*.py`): используют MockRepo из `ai_mini_box.tests.mocks`. Не требуют БД.
2. **Интеграционные тесты** (`tests/integration/test_*.py`): используют CliRunner, проверяют CLI-команды.
3. **Каждый тест** должен быть независимым (не полагаться на порядок запуска).
4. **Не тестировать внешние API** (Telegram, Email, LLM) — только моки.
5. **Добавить тесты в CI до мержа** — PR без зелёных тестов не принимается.
